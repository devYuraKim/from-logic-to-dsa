### 6. 피보나치 함수 구현

그래서 면접관이 **피보나치 수열**을 설명해주고 (고등학교 때 본 이후로 기억에 없었음), 이걸 코드로 구현하면 어떻게 될까요? 함.

구두로 설명하면 너무 예쁘게 나오는데, 코드로 구현하려니까 내가 내 코드가 통제가 안 됨. 그래서 면접관이 'sum을 더해야 할까요? 아니면 다른 값으로 관리해야 할까요?' 했을 때 문제가 뭐였는지 몰랐음. 난 이전값을 더하는 게 아니라 그때까지 더한 모든 값을 더하고 있었던 거임.

```text
fib(n + 2) = fib(n + 1) + fib(n)
fib(0) = fib(1) = 1

f(2) = f(1) + f(0) = 2
f(3) = f(2) + f(1) = 3
f(4) = f(3) + f(2) = 3 + 2 = 5
```

```java
Integer fib(Interger n) {     //3

	Integer value;
	Integer sum = 0;

    for ( int i=0; int i < n; i++){   // i가 0일 때, 2 < 3 그만 두세요   // i=0, n=3 > i=1, n=3 > i=2, n=3

	    if ( i = 0 | i = 1 ) {
            value = 1; // i=0 -> value = 1, i=1 value = 1
        } else {
	        value = sum; // fib(n + 2) = fib(n + 1) + fib(n)
        }

        value = sum
	    sum = sum + value; // 0+1 -> 1, 1+1 = 2
    }
}
```

## 2. Self-Analysis: 강점과 약점

### I. 잘했다고 생각하는 점

1. 포기하지 않았음. 늘 자료구조/알고리듬 전혀 몰라도 맨땅에 헤딩해서 해법 자료구조나 알고리듬 초입에는 접근함. 저번에는 PQ 몰랐고, 이번에는 DP 몰랐음. 그래도 함.
2. 방향이 안 잡힐 때도 think aloud 하면서 재가 계속 생각 중이라는 걸 공유함.
3. 멍하니 있지 않고 예제 넣어서 brute force라도 해 봄.
4. 문제 해결 못했어도 면접관이 언급한 성능 문제 해결법 질문하고, memoization이 웹개발 중 React에도 있는 개념이라고 연결함.

### II. 아쉽다고 생각하는 점

1. <mark>**Java 기본 문법**</mark> 왜 저 모양임? <mark>**for loop, if-else, 값 비교, 논리 연산자**</mark> 정도는 틀리지 않아야 하는 거 아님? 어떻게 Java가 주력 언어가 아닌 사람이 나보다 더 정확하게 씀? 말이나 되는 일임? 그리고 <mark>**for loop 종료조건**</mark> 왜 이렇게 헷갈려함... $i<n$인지 $i<(n+1)$인지 너무 헷갈림... 왜 이 모양임...

2. **<mark>자료형</mark> 고려 못함**: 자료구조만 고민하면 뭘함. 자료형까지 고민해야지.

3. <mark>**예제에 함몰**</mark>: 이러면 안 됨. 예제는 가장 단순한 케이스를 알려주는 거라는 걸 늘 기억해야 함. 그런데 문제 풀이 경험이 없고, 자료구조나 알고리듬도 모르다보니 우선 brute force가 용이한 예제에 집착하는 경향이 있음.

4. **가지고 있는 도구(자료구조/알고리듬)가 부족함**. 그래서 논리 자체는 접근하는데, 그걸 코드로 구현해낼 수가 없음. 면접관이 '$2^n$은 어떻게 풀어나갈까요?' 같은 힌트를 줘도 아는 게 없으니까 접근법은 맞는데, 구현이 안 됨.

5. <mark>**상태(State) 정의 없이 코딩을 시작**</mark>: 문제를 풀 때 방향은 맞지만, **'무엇을 기억해야 하는지/무엇을 구하려고 하는지'** 정의하지 않고 코딩에 들어감. 그래서 **'코드를 치면서 생각을 정리 > 되는 부분부터 구현 > 정작 핵심 로직(문제의 본질)은 놓치고 변죽만 울림'**.

   예1) 피보나치 수열  
   실제 필요한 것은 "**이전 두 값**"인데 이를 정의하지 않아서 "**전체 합을 누적**"하는 방식으로 구현함. (불필요한 상태 관리)

   예2) Subset Sum  
   "**특정 합을 만들 수 있는지**"를 기준으로 상태를 관리해야 하는데, 숫자를 빼보는 방식으로 접근하다가 막힘. (조합 문제를 순서 문제로 접근)  
   (그런데 뭐, 이건 이거 아니었으면 계속 silence였을 거라서 brute force라도 해보면서 면접관에게 뭐라도 해보려는 모습을 보여줘야 했음)

   > <mark>“**알고리즘 이름을 몰라도 문제의 구조를 인식할 수 있어야 한다**”</mark>  
   > 지금 접근은 각 원소를 빼보는 방식인데, 이건 순서 기반이라 조합 문제를 제대로 다루지 못하는 것 같습니다.  
   > 모든 subset을 고려해야 할 것 같은데, 그 경우 $2^n$이 되어 비효율적입니다.  
   > 이걸 줄이기 위해 어떤 상태를 저장하는 방식으로 가야 할 것 같은데, 지금 제가 그 부분을 정확히 정의하지 못하고 있습니다.

## 3. Action Plan: Engineering 근육 형성을 위한 로드맵

### 1. Java 기초 문법 코드 카타

> 목표: 참고 자료 없이 5분 안에 타이핑할 수 있을 때까지 반복

- **For Loop**: `for (int i = 0; i < n; i++) { ... }` — 0부터 시작, n 미만까지
- **조건문 + 논리 연산자**: `if (a == b && c != d) || (e > f)` — 비교/논리 연산자 구분
- **자료형**: 합계·카운트 변수는 무조건 `long`으로 시작

### 2. 상태 기억하기 연습 — 피보나치 완파

> DP = "방금 계산한 거 어딘가에 적어두고 다시 쓰는 것"

- **Step A**: 재귀(Recursion)로 피보나치 구현 → 계산량이 폭발하는 것 확인
- **Step B**: `long[] memo` 배열 만들어서 계산된 값 저장, 값 있으면 바로 반환
- **Step C**: For loop만으로 배열을 앞에서부터 채워나가는 방식 구현

**DP 대표 5개** (이 순서로):

1. Fibonacci
2. Climbing Stairs
3. Coin Change
4. Subset Sum (LeetCode 416)
5. Knapsack

### 3. Modulo와 Early Return 습관화

> 목표: 문제 보자마자 "이건 절대 안 돼"라고 말할 수 있는 조건 먼저 찾기

오늘의 교훈:

```java
if (totalSum % 2 != 0) return false;
```

### 4. State 정의 연습

> 목표: 코딩 시작 전 `dp[i]`가 뭔지 한 줄로 먼저 쓰기

| 문제            | 상태 정의                                    |
| --------------- | -------------------------------------------- |
| Fibonacci       | `dp[i]` = i번째 피보나치 값                  |
| Climbing Stairs | `dp[i]` = i번째 계단까지 가는 방법 수        |
